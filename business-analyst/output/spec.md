# Спецификация улучшения системы классификации запросов
## Мультиагентная система обслуживания клиентов COMFY

### Контекст проекта
Система состоит из 4 AI-агентов: Orchestrator (маршрутизатор), FAQ, Customers, Product. Основная проблема: неправильная маршрутизация запросов на подбор товаров из-за несовершенной логики классификации в orchestrator.

---

## US-001: Анализ текущей логики классификации

### Спецификация
Провести детальный анализ существующей логики классификации в orchestrator для выявления причин неправильной маршрутизации запросов о подборе товаров. Анализ должен включать изучение промптов, ключевых слов, алгоритмов принятия решений и статистики ошибок.

### Acceptance Criteria

**Дано:** Текущая система классификации в orchestrator  
**Когда:** Выполняется анализ логики классификации  
**Тогда:** 
- Создан отчет с выявленными проблемами классификации
- Определены конкретные случаи неправильной маршрутизации
- Проанализированы промпты и ключевые слова
- Выявлены паттерны ошибок в логах

**Дано:** Запросы пользователей на подбор товаров  
**Когда:** Анализируются исторические данные  
**Тогда:**
- Собрана статистика по типам неправильно классифицированных запросов
- Определен процент ошибок маршрутизации для product-agent
- Выявлены наиболее проблемные категории запросов

### Edge Cases
- Запросы с неоднозначной формулировкой
- Смешанные запросы (FAQ + подбор товаров)
- Запросы на языках, отличных от основного
- Очень короткие или очень длинные запросы

### Ограничения и допущения
- Анализ ограничен доступными логами за последние 30 дней
- Предполагается наличие логирования решений классификатора
- Анализ проводится только для текстовых запросов

---

## US-002: Улучшение промпта классификатора

### Спецификация
Модифицировать промпт классификатора в orchestrator для корректного распознавания запросов на подбор товаров независимо от упоминания конкретных категорий. Промпт должен учитывать различные формулировки пользователей и контекст запроса.

### Acceptance Criteria

**Дано:** Запрос пользователя на подбор товара без указания конкретной категории  
**Когда:** Классификатор обрабатывает запрос с новым промптом  
**Тогда:**
- Запрос корректно маршрутизируется в product-agent
- Точность классификации для запросов подбора товаров увеличивается до 90%
- Время обработки запроса не увеличивается более чем на 100мс

**Дано:** Запросы с описанием характеристик товара  
**Когда:** Используется обновленный промпт  
**Тогда:**
- Классификатор распознает намерение подбора товара
- Запрос направляется в product-agent независимо от отсутствия названия категории
- Сохраняется контекст для последующей обработки

### Edge Cases
- Запросы с негативными формулировками ("не хочу дорогой телефон")
- Сравнительные запросы ("что лучше iPhone или Samsung")
- Запросы с условиями ("если есть в наличии")
- Запросы с бюджетными ограничениями

### Ограничения и допущения
- Промпт должен работать с моделями AWS Bedrock (Nova Micro, Claude)
- Максимальная длина промпта ограничена возможностями модели
- Промпт должен поддерживать украинский и русский языки

## US-003: Добавление ключевых слов для product-agent

### Спецификация
Расширить и оптимизировать список ключевых слов и фраз для маршрутизации запросов в product-agent. Включить различные варианты формулировок запросов на подбор товаров, синонимы и сленговые выражения.

### Acceptance Criteria

**Дано:** Расширенный список ключевых слов для product-agent  
**Когда:** Пользователь использует различные формулировки для подбора товаров  
**Тогда:**
- Система распознает минимум 50 различных вариантов запросов на подбор
- Включены синонимы и альтернативные формулировки
- Добавлены сленговые и разговорные выражения
- Учтены региональные особенности языка

**Дано:** Запросы с характеристиками товаров  
**Когда:** Используются технические термины или описания  
**Тогда:**
- Система распознает технические характеристики как индикаторы подбора
- Включены термины из различных категорий товаров
- Обработаны сокращения и аббревиатуры

### Edge Cases
- Опечатки в ключевых словах
- Использование эмодзи в запросах
- Смешение языков в одном запросе
- Устаревшие названия товаров или брендов

### Ограничения и допущения
- Список ключевых слов хранится в конфигурационном файле
- Поддержка только украинского и русского языков
- Максимум 500 ключевых слов для оптимальной производительности

## US-004: Создание тестовых сценариев

### Спецификация
Разработать комплексный набор тестовых сценариев для проверки корректности классификации запросов на подбор товаров. Сценарии должны покрывать различные варианты формулировок, edge cases и граничные условия.

### Acceptance Criteria

**Дано:** Набор тестовых сценариев для классификации  
**Когда:** Выполняется тестирование системы классификации  
**Тогда:**
- Создано минимум 100 тестовых случаев
- Покрыты все основные категории товаров COMFY
- Включены позитивные и негативные тест-кейсы
- Добавлены граничные случаи и edge cases

**Дано:** Различные варианты формулировок запросов  
**Когда:** Запускаются автоматизированные тесты  
**Тогда:**
- Тесты выполняются автоматически через API
- Результаты сохраняются в структурированном формате
- Генерируется отчет о покрытии тестами
- Время выполнения полного набора тестов не превышает 10 минут

### Edge Cases
- Пустые запросы
- Запросы только из специальных символов
- Очень длинные запросы (>1000 символов)
- Запросы на неподдерживаемых языках

### Ограничения и допущения
- Тесты выполняются в тестовой среде
- Используются моковые данные для товаров
- Тестирование ограничено функциональностью классификации

## US-005: Логирование решений классификатора

### Спецификация
Реализовать детальное логирование всех решений классификатора для отслеживания и анализа случаев неправильной маршрутизации. Логи должны содержать достаточно информации для последующего анализа и улучшения системы.

### Acceptance Criteria

**Дано:** Запрос пользователя к системе  
**Когда:** Классификатор принимает решение о маршрутизации  
**Тогда:**
- Логируется исходный запрос пользователя
- Сохраняется решение классификатора с уровнем уверенности
- Записывается время обработки запроса
- Фиксируется выбранный агент и причина выбора

**Дано:** Система логирования решений  
**Когда:** Накапливаются данные о классификации  
**Тогда:**
- Логи структурированы в JSON формате
- Включены метаданные (timestamp, session_id, user_id)
- Логи ротируются ежедневно
- Размер лог-файла не превышает 100MB

### Edge Cases
- Ошибки в процессе логирования
- Недоступность системы логирования
- Переполнение дискового пространства
- Конфиденциальная информация в запросах

### Ограничения и допущения
- Логи хранятся локально на сервере orchestrator
- Персональные данные маскируются или исключаются
- Логирование не должно влиять на производительность более чем на 5%

---

## US-006: Правильная маршрутизация запросов на подбор товаров

### Спецификация
Обеспечить корректную маршрутизацию пользовательских запросов на подбор товаров в product-agent для получения релевантных рекомендаций. Система должна распознавать намерения пользователя независимо от формулировки запроса.

### Acceptance Criteria

**Дано:** Пользователь отправляет запрос на подбор товара с описанием характеристик  
**Когда:** Система обрабатывает запрос  
**Тогда:**
- Запрос маршрутизируется в product-agent
- Product-agent возвращает релевантные рекомендации товаров
- Время ответа не превышает 3 секунд
- Пользователь получает минимум 3 варианта товаров

**Дано:** Различные формулировки запросов на подбор  
**Когда:** Пользователь использует разные способы описания потребностей  
**Тогда:**
- Система корректно интерпретирует намерение в 90% случаев
- Маршрутизация происходит независимо от упоминания категорий
- Сохраняется контекст для уточняющих вопросов
- Обеспечивается персонализация рекомендаций

### Edge Cases
- Запросы с противоречивыми требованиями
- Запросы на несуществующие товары
- Запросы с неполной информацией
- Запросы на товары вне ассортимента COMFY

### Ограничения и допущения
- Product-agent имеет актуальную базу товаров
- Система работает только с товарами из каталога COMFY
- Рекомендации основаны на доступных характеристиках товаров

---

## US-007: Fallback механизм для неопределенных запросов

### Спецификация
Реализовать fallback механизм для случаев, когда классификатор не может с достаточной уверенностью определить подходящего агента. Механизм должен минимизировать ошибки маршрутизации и обеспечивать приемлемый пользовательский опыт.

### Acceptance Criteria

**Дано:** Запрос с низким уровнем уверенности классификации (<70%)  
**Когда:** Классификатор не может определить подходящего агента  
**Тогда:**
- Активируется fallback механизм
- Пользователю предлагается уточнить запрос
- Показываются варианты возможных категорий помощи
- Запрос логируется для последующего анализа

**Дано:** Неоднозначный запрос пользователя  
**Когда:** Запрос может относиться к нескольким агентам  
**Тогда:**
- Система запрашивает уточнение у пользователя
- Предлагаются конкретные варианты выбора
- Сохраняется контекст для последующих запросов
- Время ожидания ответа пользователя ограничено 5 минутами

### Edge Cases
- Полностью неопределенные запросы
- Запросы на неподдерживаемых языках
- Технические ошибки в процессе классификации
- Превышение лимитов API моделей

### Ограничения и допущения
- Fallback срабатывает при уверенности классификатора <70%
- Максимум 3 попытки уточнения запроса
- После неудачных попыток запрос передается в FAQ-agent

---

## US-008: Мониторинг качества классификации

### Спецификация
Внедрить систему мониторинга и метрик для отслеживания качества классификации запросов. Система должна предоставлять данные для принятия решений об улучшении алгоритмов маршрутизации.

### Acceptance Criteria

**Дано:** Система мониторинга качества классификации  
**Когда:** Накапливаются данные о работе классификатора  
**Тогда:**
- Рассчитывается точность классификации по каждому агенту
- Отслеживается количество fallback случаев
- Измеряется время обработки запросов
- Генерируются ежедневные отчеты

**Дано:** Метрики качества классификации  
**Когда:** Анализируется производительность системы  
**Тогда:**
- Доступны метрики precision, recall, F1-score для каждого агента
- Отслеживается тренд изменения качества во времени
- Выявляются аномалии в работе классификатора
- Настроены алерты при снижении качества ниже 85%

### Edge Cases
- Недостаток данных для расчета метрик
- Технические сбои в системе мониторинга
- Изменение паттернов пользовательских запросов
- Сезонные колебания в типах запросов

### Ограничения и допущения
- Метрики рассчитываются на основе логов за последние 7 дней
- Система мониторинга не влияет на производительность основной системы
- Отчеты генерируются автоматически в 00:00 UTC

## US-009: Регрессионное тестирование

### Спецификация
Провести комплексное регрессионное тестирование всех типов запросов для проверки того, что внесенные изменения не нарушили существующую функциональность системы маршрутизации.

### Acceptance Criteria

**Дано:** Обновленная система классификации  
**Когда:** Выполняется регрессионное тестирование  
**Тогда:**
- Протестированы все существующие сценарии маршрутизации
- Проверена работа всех четырех агентов
- Подтверждена корректность маршрутизации FAQ запросов
- Проверена работа с данными клиентов через customers-agent

**Дано:** Набор регрессионных тестов  
**Когда:** Тесты выполняются автоматически  
**Тогда:**
- Все существующие тесты проходят успешно
- Новые тесты для подбора товаров также проходят
- Производительность системы не ухудшилась
- Отсутствуют критические ошибки в логах

### Edge Cases
- Одновременная нагрузка на все агенты
- Сбои в работе внешних сервисов (MongoDB, Redis)
- Превышение лимитов AWS Bedrock
- Потеря соединения WebSocket

### Ограничения и допущения
- Тестирование проводится в изолированной тестовой среде
- Используются тестовые данные, не влияющие на продакшн
- Регрессионные тесты выполняются перед каждым релизом

## Общие ограничения проекта

### Технические ограничения
- Система работает на базе AWS Bedrock с моделями Nova Micro и Claude
- Используется архитектура микросервисов с портами 8000-8003
- Ограничения по производительности: время ответа <3 сек, throughput >100 req/min
- Поддержка только украинского и русского языков

### Бизнес-ограничения
- Бюджет проекта ограничен текущими ресурсами команды
- Сроки реализации: 4 недели для критических US (high priority)
- Изменения не должны нарушать работу существующих интеграций
- Требуется обратная совместимость с текущими API

### Допущения
- Каталог товаров COMFY остается стабильным в период разработки
- Пользователи используют стандартные формулировки запросов
- Система мониторинга AWS доступна для отслеживания метрик
- Команда имеет доступ к историческим данным запросов пользователей
