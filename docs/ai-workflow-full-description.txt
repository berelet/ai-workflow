================================================================================
ПОВНИЙ ТЕХНІЧНИЙ ТА БІЗНЕС-ОПИС СИСТЕМИ AI WORKFLOW (VISUAL PIPELINE BUILDER)
================================================================================

Цей документ містить деталізовану інформацію про архітектуру, логіку та реалізацію 
системи для підготовки професійної презентації.

--------------------------------------------------------------------------------
СТРУКТУРА ПРЕЗЕНТАЦІЇ (ПО СЛАЙДАХ)
--------------------------------------------------------------------------------

СЛАЙД 1: Титульний — Візія проекту
• Назва: AI Workflow: Екосистема автономної агентної розробки.
• Суть: Платформа для візуального проектування та виконання складних SDLC-процесів 
  (Software Development Life Cycle) за допомогою мережі спеціалізованих AI-агентів.
• Головна мета: Перехід від лінійних скриптів до гнучких візуальних графів виконання.

СЛАЙД 2: Проблема та Контекст (Business Case)
• Проблема: Традиційні AI-системи мають жорстку логіку 1:1 (один проект — один сценарій).
• Обмеження: Відсутність візуального контролю, складність налаштування перевірок (Review), 
  низька прозорість процесу для менеджера.
• Рішення: Перехід до архітектури 1:N, де для одного проекту можна створити безліч 
  спеціалізованих пайплайнів (Discovery, Hotfix, Full Feature, Refactoring).

СЛАЙД 3: Архітектура "Double Pass" (Золотий стандарт якості)
• Концепція: Жодна дія не приймається без верифікації.
• Вузол "Виконавець" (Agent Node): Генерує артефакт (код, вимоги, дизайн).
• Вузол "Ревізор" (Reviewer Node): Перевіряє артефакт на основі суворих правил (Skills).
• Результат: Мінімізація "галюцинацій" AI та гарантована відповідність Acceptance Criteria.

СЛАЙД 4: Visual Pipeline Builder (Інтерфейс управління)
• Технологія: Drawflow (Vanilla JS) — легкий та швидкий двигун для роботи з графами.
• Можливості:
  - Drag-and-drop агентів на полотно.
  - Візуальне з'єднання етапів (Edges).
  - Контекстне налаштування кожного вузла (Custom Prompt + Skills).
• Бізнес-цінність: Оператор бачить "карту польоту" завдання в реальному часі.

СЛАЙД 5: Технічний стек (The Engine)
• Backend: Python (FastAPI) для API та логіки, Bash/tmux для сесійної ізоляції.
• Frontend: Vanilla JS/HTML/CSS (без важких фреймворків для швидкості).
• Orchestration: Власний cli.py, що керує tmux-сесіями та ін'єкцією контексту.
• AI Engine: Підтримка сучасних LLM (Claude 3.5 Sonnet, GPT-4o) через уніфікований інтерфейс.

СЛАЙД 6: Система скілів та Tessl Tiles
• Що таке Skill: Документ SKILL.md, що описує стандарти (наприклад, "Security Best Practices" 
  або "React 19 Patterns").
• Динамічна ін'єкція: Скіли підключаються ТІЛЬКИ до ревізорів, що робить перевірку 
  максимально фокусованою та професійною.
• Бібліотека Tessl: Можливість перевикористання експертних знань у різних проектах.

СЛАЙД 7: Життєвий цикл задачі (Data Flow)
1. Backlog: Поява задачі в системі.
2. Decomposition: PM розбиває її на User Stories.
3. Execution: Проходження по візуальному графу (BA -> DEV -> QA).
4. Artifacts: Збереження проміжних результатів у директорії `.ai-workflow/artifacts/`.
5. Validation: Автоматичні тести та ручне підтвердження.

--------------------------------------------------------------------------------
ГЛИБОКІ ТЕХНІЧНІ ДЕТАЛІ (ДЛЯ Q&A СЕСІЇ)
--------------------------------------------------------------------------------

1. Управління контекстом (Prompt Engineering):
Система не просто надсилає запит до AI. Вона збирає "Context Package", куди входять:
- `project.md`: Глобальні цілі та стек.
- `backlog.json`: Поточний стан усіх завдань.
- `pipeline.md`: Історія проходження етапів по поточній задачі.
Це забезпечує консистентність: розробник знає, що написав аналітик, а тестер знає, 
що реалізував розробник.

2. Ізоляція процесів (tmux integration):
Кожен проект запускається в окремому вікні tmux. Це дозволяє:
- Одночасно працювати над декількома проектами.
- Зберігати стан сесії навіть при розриві з'єднання.
- Легко перемикатися між режимом "AI працює автономно" та "Я втручаюся в процес".

3. Структура даних (The .ai-workflow folder):
Система зберігає все в проектній папці, що робить її Git-friendly:
- `pipeline-config.json`: Опис структури етапів.
- `backlog.json`: База даних завдань.
- `artifacts/`: Повна історія змін для кожної задачі (диференційні зміни коду, логи тестів).

4. Гнучкість пайплайнів:
Завдяки переходу на графову структуру, система підтримує:
- Branching (Розгалуження): Наприклад, паралельний запуск розробки фронтенду та бекенду.
- Loops (Цикли): Автоматичне повернення від QA до DEV при виявленні багів.
- Conditional Logic: Виконання певних вузлів лише при виконанні умов.
