# Специфікація: Рефакторинг JWT-аутентифікації

## Загальний опис
Рефакторинг системи JWT-аутентифікації для усунення security vulnerability в middleware/auth.py та дублювання коду між модулями аутентифікації.

## US-15.1: Реалізація JWT-валідації в middleware

### Специфікація
Замінити TODO-заглушку в `middleware/auth.py` на повноцінну JWT-валідацію з використанням існуючої логіки з `utils/jwt_helper.py`.

**Технічні зміни:**
- Видалити заглушку `return token` з функції валідації
- Імплементувати `jwt.decode()` з перевіркою підпису та expiration
- Обробити `ExpiredSignatureError` та `InvalidTokenError`
- Витягнути `user_id` з `payload["sub"]`

### Acceptance Criteria

**AC-15.1.1: JWT-підпис валідується**
- Дано: Запит з валідним JWT токеном
- Коли: Middleware обробляє токен
- Тоді: Токен успішно декодується через `jwt.decode()`

**AC-15.1.2: Прострочені токени відхиляються**
- Дано: Запит з простроченим JWT токеном
- Коли: Middleware обробляє токен
- Тоді: Повертається HTTP 401 з `ExpiredSignatureError`

**AC-15.1.3: Невалідні токени відхиляються**
- Дано: Запит з пошкодженим/невалідним JWT токеном
- Коли: Middleware обробляє токен
- Тоді: Повертається HTTP 401 з `InvalidTokenError`

**AC-15.1.4: User ID витягується коректно**
- Дано: Запит з валідним JWT токеном
- Коли: Middleware декодує токен
- Тоді: `user_id` витягується з `payload["sub"]`

### Edge Cases
- Токен без `sub` поля
- Токен з некоректним форматом `sub`
- Відсутній токен в заголовку
- Токен з некоректним алгоритмом підпису

### Обмеження
- Використовувати існуючий SECRET_KEY з конфігурації
- Зберегти існуючу сигнатуру функцій
- Не змінювати HTTP статус коди

## US-15.2: Усунення дублювання коду

### Специфікація
Централізувати JWT-валідацію в `dependencies.py` та створити реекспорт в `middleware/auth.py` для зворотної сумісності.

**Технічні зміни:**
- Винести спільну JWT-логіку в `dependencies.py`
- Створити функцію `_validate_jwt_token()` в `dependencies.py`
- Реекспортувати необхідні функції в `middleware/auth.py`
- Видалити дублюючий код з `middleware/auth.py`

### Acceptance Criteria

**AC-15.2.1: Єдина точка JWT-валідації**
- Дано: Система з рефакторингом
- Коли: Перевіряється код JWT-валідації
- Тоді: Логіка decode/verify існує тільки в `dependencies.py`

**AC-15.2.2: Зворотна сумісність імпортів**
- Дано: Існуючі імпорти з `middleware/auth.py`
- Коли: Код використовує ці імпорти
- Тоді: Всі імпорти працюють без змін

**AC-15.2.3: Відсутність дублювання**
- Дано: Рефакторений код
- Коли: Аналізується JWT-логіка
- Тоді: Немає дублювання між `middleware/auth.py` та `dependencies.py`

### Edge Cases
- Циклічні імпорти між модулями
- Зміна сигнатури функцій при реекспорті
- Конфлікти імен при реекспорті

### Обмеження
- Зберегти всі існуючі імпорти
- Не змінювати публічний API
- Мінімізувати зміни в існуючому коді

---

## US-15.3: Перевірка використання JWT-валідації

### Специфікація
Аудит всіх роутерів для підтвердження використання повноцінної JWT-валідації з `dependencies.py`.

**Технічні зміни:**
- Перевірити імпорти в усіх 11 роутерах
- Підтвердити використання `get_current_user_id` з `dependencies.py`
- Перевірити admin endpoints на використання `require_admin`
- Перевірити optional auth endpoints

### Acceptance Criteria

**AC-15.3.1: Роутери використовують dependencies.py**
- Дано: Всі 11 роутерів
- Коли: Аналізуються імпорти аутентифікації
- Тоді: Всі імпортують `get_current_user_id` з `dependencies.py`

**AC-15.3.2: Відсутні прямі імпорти middleware**
- Дано: Всі файли роутерів
- Коли: Перевіряються імпорти
- Тоді: Жоден файл не імпортує напряму з `middleware/auth.py`

**AC-15.3.3: Admin endpoints захищені**
- Дано: Admin endpoints
- Коли: Перевіряється аутентифікація
- Тоді: Використовується `require_admin` з повною JWT-перевіркою

**AC-15.3.4: Optional auth працює**
- Дано: Endpoints з optional auth
- Коли: Перевіряється логіка
- Тоді: Використовується JWT decode з `dependencies.py`

### Edge Cases
- Роутери з кастомною аутентифікацією
- Публічні endpoints без аутентифікації
- Middleware endpoints з особливою логікою

### Обмеження
- Не змінювати логіку роутерів
- Зберегти існуючу функціональність
- Мінімальні зміни в імпортах

## US-15.4: Верифікація працездатності

### Специфікація
Комплексне тестування системи аутентифікації після рефакторингу для підтвердження збереження функціональності.

**Технічні перевірки:**
- Старт сервера без помилок
- Тестування всіх auth endpoints
- Перевірка валідації токенів
- Тестування error handling

### Acceptance Criteria

**AC-15.4.1: Сервер стартує успішно**
- Дано: Рефакторений код
- Коли: Запускається FastAPI сервер
- Тоді: Сервер стартує без помилок, HTTP 200 на `/`

**AC-15.4.2: Login працює**
- Дано: Валідні credentials
- Коли: POST запит на `/api/v2/auth/login`
- Тоді: Відправляється код верифікації, HTTP 200

**AC-15.4.3: Verify повертає токени**
- Дано: Валідний код верифікації
- Коли: POST запит на `/api/v2/auth/verify`
- Тоді: Повертаються access та refresh токени, HTTP 200

**AC-15.4.4: /me працює з валідним токеном**
- Дано: Валідний access токен
- Коли: GET запит на `/api/v2/auth/me`
- Тоді: Повертаються дані користувача, HTTP 200

**AC-15.4.5: /me відхиляє невалідні токени**
- Дано: Невалідний/прострочений токен
- Коли: GET запит на `/api/v2/auth/me`
- Тоді: Повертається HTTP 401

**AC-15.4.6: Refresh оновлює токен**
- Дано: Валідний refresh токен
- Коли: POST запит на `/api/v2/auth/refresh`
- Тоді: Повертається новий access токен, HTTP 200

**AC-15.4.7: Logout працює**
- Дано: Валідний токен
- Коли: POST запит на `/api/v2/auth/logout`
- Тоді: Токен інвалідується, HTTP 200

### Edge Cases
- Одночасні запити з одним токеном
- Токени з різними алгоритмами
- Запити без заголовків аутентифікації
- Некоректні JSON payload

### Обмеження
- Тестування на development середовищі
- Використання існуючої тестової БД
- Збереження rate limiting налаштувань

## Загальні обмеження та допущення

### Обмеження
- Не змінювати існуючі API endpoints
- Зберегти зворотну сумісність
- Мінімізувати зміни в існуючому коді
- Не впливати на продуктивність

### Допущення
- SECRET_KEY налаштований коректно
- PostgreSQL та Redis доступні
- Існуючі тести покривають основну функціональність
- Rate limiting налаштування не змінюються
